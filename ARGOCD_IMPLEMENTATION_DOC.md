# ArgoCD Implementation Documentation

This document outlines the GitOps architecture and deployment strategy using ArgoCD for the Banking Application. It covers how ArgoCD manages our Kubernetes manifests and Helm charts stored in the `k8s-manifests` repository.

## Overview

We use **ArgoCD** as our declarative, GitOps continuous delivery tool for Kubernetes. 
Instead of pushing changes to our cluster via a CI pipeline, the CI pipeline simply updates the `k8s-manifests` repository (e.g., updating an image tag). ArgoCD continuously monitors this repository and pulls changes, ensuring the cluster's state matches the state defined in Git.

## Repository Structure

Our deployment source of truth is the `k8s-manifests` repository, which is organized as follows:
* `charts/`: Contains the Helm charts for all microservices (e.g., `account-service`, `fraud-detection-service`, `frontend`).
* `infra/`: Contains raw Kubernetes manifests for foundational infrastructure (e.g., namespaces, databases, messaging queues, monitoring).

## 1. ArgoCD Installation & Setup

If ArgoCD is not yet installed in the cluster, install it using the following commands:
```bash
kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
```

To access the ArgoCD UI, port-forward the service:
```bash
kubectl port-forward svc/argocd-server -n argocd 8080:443
```
*(The initial admin password is autogenerated and stored in the `argocd-initial-admin-secret` secret).*

## 2. Connect the Git Repository

ArgoCD needs access to the `k8s-manifests` repository. If the repository is private, you must add the repository credentials via the ArgoCD UI or CLI:
```bash
argocd repo add https://github.com/your-org/k8s-manifests.git --username <username> --password <token>
```

## 3. Implementing ArgoCD Applications

We define our deployments as ArgoCD `Application` Custom Resources. This allows us to manage ArgoCD configurations declaratively.

### A. Deploying Infrastructure (Raw YAMLs)

For directories containing standard Kubernetes manifests (like `infra/data` or `infra/gateway`), we use a standard directory-based Application.

**Example `infra-app.yaml`:**
```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: banking-infrastructure
  namespace: argocd
spec:
  project: default
  source:
    repoURL: 'https://github.com/your-org/k8s-manifests.git'
    targetRevision: main
    path: infra
    directory:
      recurse: true # Deploys all manifests in subdirectories
  destination:
    server: 'https://kubernetes.default.svc'
    namespace: banking-system
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

### B. Deploying Microservices (Helm Charts)

For the microservices located in the `charts/` directory, ArgoCD natively supports Helm. It will run `helm template` and apply the generated manifests.

**Example `account-service-app.yaml`:**
```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: account-service
  namespace: argocd
spec:
  project: default
  source:
    repoURL: 'https://github.com/your-org/k8s-manifests.git'
    targetRevision: main
    path: charts/account-service
    helm:
      valueFiles:
        - values.yaml
  destination:
    server: 'https://kubernetes.default.svc'
    namespace: banking-system
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

## 4. Advanced Pattern: ApplicationSets (App of Apps)

To avoid managing dozens of individual `Application` manifests for our 10+ microservices, we utilize an **ApplicationSet** (or the "App of Apps" pattern). This automatically discovers Helm charts in the `charts/` directory and creates an ArgoCD Application for each one.

**Example `microservices-appset.yaml`:**
```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: banking-microservices
  namespace: argocd
spec:
  generators:
    - git:
        repoURL: https://github.com/your-org/k8s-manifests.git
        revision: main
        directories:
          - path: charts/*
  template:
    metadata:
      name: '{{path.basename}}'
    spec:
      project: default
      source:
        repoURL: https://github.com/your-org/k8s-manifests.git
        targetRevision: main
        path: '{{path}}'
        helm:
          valueFiles:
            - values.yaml
      destination:
        server: https://kubernetes.default.svc
        namespace: banking-system
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
```

## 5. Sync Policy and Automated Deployments

All our applications are configured with automated sync policies:
* **Automated Sync**: ArgoCD pulls the Git repository every 3 minutes. If the desired state (Git) differs from the live state (Cluster), ArgoCD applies the changes.
* **Prune**: If a file is deleted from Git, ArgoCD will delete the corresponding resource from the cluster.
* **Self-Heal**: If a resource is manually modified via `kubectl` directly on the cluster, ArgoCD will automatically revert it to the state defined in Git.

## Summary of CI/CD Flow with ArgoCD
1. Developer pushes application code.
2. GitHub Actions CI pipeline runs (Tests, Code Quality, Docker Build & Push).
3. The final CI step updates the `values.yaml` in the `k8s-manifests` repository with the new Docker image tag.
4. ArgoCD detects the commit in `k8s-manifests`.
5. ArgoCD synchronizes the cluster, pulling the new image and deploying the update with zero downtime.
